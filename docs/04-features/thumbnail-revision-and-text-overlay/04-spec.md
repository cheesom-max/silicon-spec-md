# 생성본 수정과 문구 오버레이 스펙

## 목적
- 이미 생성된 썸네일을 AI로 다시 수정하고, 이후 사용자가 직접 문구를 얹는 기능을 정의한다.

## 사용 순서
- 4단계: 수정 또는 편집 기능 구현 전에 읽고 갱신한다.
- 수정 비용, 버전 관리, 문구 오버레이 범위가 바뀔 때 수정한다.

## 대상 독자
- 수정 요청, 결과 버전 관리, 텍스트 오버레이 UI를 구현하고 검토하는 사람.

## 범위 밖
- 범용 이미지 편집기.
- 외부 이미지 업로드를 이용한 수정.
- 결제 적립 상세.

## 정본으로 다루는 범위
- 기존 생성본 수정 규칙.
- 수동 문구 오버레이 규칙.
- 10 크레딧 차감 조건.
- 수용 기준과 검증 계획.

## 마지막 업데이트 시점
- 수정 비용, 문구 오버레이 범위, 버전 저장 방식이 바뀔 때.

## 정본 링크
- 제품 의도는 `docs/01-product.md`를 참고한다.
- 아키텍처 맥락은 `docs/02-system-architecture.md`를 참고한다.
- 크레딧 정책은 `docs/06-adrs/06-ADR-0001-credit-ledger-and-deduction-policy.md`를 참고한다.

## 문제
- 첫 생성 결과가 항상 바로 만족스럽지는 않다. 사용자는 같은 맥락을 유지한 채 이미지를 한 번 더 다듬고 싶어 한다.
- 동시에 최종 문구는 직접 제어하고 싶어 하므로, AI 수정과 텍스트 입력은 분리된 흐름이어야 한다.

## 고객 서사
- 사용자는 생성 이력에서 기존 썸네일을 선택한다.
- 원하는 수정 내용을 자연어로 입력하고 다시 생성한다.
- 수정본이 마음에 들면 앱 안에서 직접 문구를 얹고 최종 이미지를 다운로드한다.

## 범위
- 기존 생성본 선택.
- 자연어 수정 지시 제출.
- 수정 요청당 10 크레딧 차감.
- 수정 결과를 새 버전으로 저장.
- 단일 텍스트 오버레이 추가, 위치 조정, 색상 조정, 저장 또는 다운로드.

## 비목표
- 다중 레이어 편집.
- 마스킹, 부분 선택 편집.
- 폰트 라이브러리 확장이나 고급 타이포그래피 템플릿.

## 사용자 또는 호출자
- 로그인된 사용자 중 기존 생성 자산을 가진 사용자.

## 동작
- 사용자는 자신의 생성 이력 중 하나를 선택해 수정 요청을 보낼 수 있다.
- 수정 요청이 수락되면 잔액을 확인하고 10 크레딧을 차감한다.
- 수정 결과는 새 자산으로 저장되며 원본과 연결된 버전 체인을 남긴다.
- 수정 결과도 생성과 동일하게 무문자 검출 게이트를 통과해야 완료된다.
- 사용자는 완료된 자산 위에 직접 문구를 하나 올리고 위치, 크기, 색상 정도의 기본 속성을 조정할 수 있다.
- 텍스트 오버레이 편집은 AI 호출이 아니므로 크레딧을 차감하지 않는다.

## 수용 기준
- 다른 사용자의 자산은 수정 대상으로 선택할 수 없다.
- 수정 요청 성공 시 잔액이 정확히 10 감소하고 새 버전 자산이 저장된다.
- 수정 최종 실패 시 차감된 10 크레딧은 환불된다.
- 수동 문구 오버레이는 크레딧 차감 없이 저장 또는 다운로드할 수 있다.
- 텍스트 오버레이를 저장해도 원본 무문자 자산은 변경되지 않고 별도 최종 산출물로 남는다.

## 영향 받는 영역
- 생성 이력 UI.
- 수정 작업 상태 저장.
- 자산 버전 연결.
- 텍스트 오버레이 편집 UI.
- 크레딧 원장.

## 로컬 설계 메모
- 수정은 앱 안에서 생성된 자산만 대상으로 한다.
- 오버레이 편집은 MVP에서 단일 문구 레이어만 지원한다.
- 원본 무문자 자산과 문구가 포함된 최종 산출물은 분리 저장해 재편집을 단순하게 유지한다.

## 실행 메타데이터
- 실행 단위: 문서, 코드, 저장소 정책.
- 전제 조건: 기존 생성 자산 존재, 10 이상 잔액, 수정 어댑터, 텍스트 오버레이 UI.
- 실행 후 확인 항목: 버전 연결, 차감 이벤트, 새 자산 저장, 무료 오버레이 저장 또는 다운로드.
- 남겨야 할 증거: 수정 성공 1건에 대한 이전 자산 ID, 새 자산 ID, 차감 이벤트, 최종 산출물 메타데이터.

## 위험과 열린 질문
- 텍스트 오버레이 기능이 범용 편집기로 커질 위험이 있으므로 단일 레이어와 기본 속성만 유지한다.
- 수정 지시가 너무 짧거나 모호한 경우 품질이 낮을 수 있으나 MVP에서는 별도 프롬프트 가이드를 최소화한다.

## 검증 계획
- 소유하지 않은 자산 수정 시도가 차단되는지 확인한다.
- 수정 성공 시 10 크레딧 차감과 새 버전 저장이 일어나는지 확인한다.
- 수정 실패 시 환불이 수행되는지 확인한다.
- 텍스트 오버레이 저장과 다운로드가 무료로 수행되는지 확인한다.
