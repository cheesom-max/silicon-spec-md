# 크레딧과 결제 스펙

## 목적
- 무료 크레딧, 생성/수정 차감, Polar 기반 유료 크레딧 구매 흐름을 정의한다.

## 사용 순서
- 4단계: 크레딧이나 결제 로직 구현 전에 읽고 갱신한다.
- 가격 정책, 적립 방식, 환불 정책이 바뀔 때 수정한다.

## 대상 독자
- 크레딧 원장, 잔액 표시, 결제 시작, 웹훅 적립을 구현하고 검토하는 사람.

## 범위 밖
- Polar 내부 상품 운영 화면 구성.
- 회계 처리나 세금 상세.
- 구독제 설계.

## 정본으로 다루는 범위
- 무료 크레딧 정책.
- 생성/수정 차감 정책.
- 유료 크레딧 팩 구매 흐름.
- 수용 기준과 검증 계획.

## 마지막 업데이트 시점
- 크레딧 정책, 적립 정책, 결제 방식, 환불 정책이 바뀔 때.

## 정본 링크
- 제품 의도는 `docs/01-product.md`를 참고한다.
- 아키텍처 맥락은 `docs/02-system-architecture.md`를 참고한다.
- 크레딧 정책 ADR은 `docs/06-adrs/06-ADR-0001-credit-ledger-and-deduction-policy.md`를 참고한다.

## 문제
- 사용자는 현재 잔액과 차감 규칙을 쉽게 이해해야 한다.
- 결제가 성공했는데 잔액에 반영되지 않거나, 실패했는데 차감이 남는 상황은 신뢰를 크게 해친다.

## 고객 서사
- 사용자는 가입 직후 무료 100 크레딧으로 기능을 체험한다.
- 생성과 수정이 반복될수록 잔액이 10씩 줄어드는 것을 직관적으로 이해한다.
- 잔액이 부족하면 즉시 결제 화면으로 이동해 유료 크레딧 팩을 구매하고, 결제 완료 후 다시 작업을 이어간다.

## 범위
- 무료 100 크레딧 최초 1회 지급.
- 생성 요청당 10 크레딧 차감.
- 수정 요청당 10 크레딧 차감.
- 실패 시 환불 이벤트 생성.
- Polar 체크아웃 시작과 결제 완료 웹훅 반영.
- 현재 잔액과 최근 원장 이벤트 표시.
- 웹훅 진위 검증과 replay 방지.
- checkout 요청의 사용자, SKU, 적립량 바인딩 검증.

## 비목표
- 월 구독.
- 프로모션 코드 자체 로직.
- 수동 송금 기반 결제.

## 사용자 또는 호출자
- 로그인된 사용자.
- 서버 웹훅 처리자.

## 동작
- 사용자는 현재 잔액과 최근 적립/차감 이벤트를 볼 수 있다.
- 생성 또는 수정 요청이 수락되면 서버는 즉시 10 크레딧 차감 이벤트를 기록한다.
- 생성 또는 수정이 최종 실패하면 서버는 동일한 요청을 상쇄하는 환불 이벤트를 기록한다.
- 사용자가 유료 크레딧 팩을 선택하면 서버가 Polar 체크아웃 세션을 생성한다.
- Polar의 `order.paid` 웹훅을 받으면 서버는 해당 주문의 적립 이벤트가 이미 반영되었는지 확인한 뒤, 처음인 경우에만 크레딧을 적립한다.
- 유료 팩별 실제 적립 크레딧 수는 운영자가 관리하는 SKU-크레딧 매핑으로 결정한다.
- 웹훅 적립 전에는 Polar 서명 검증과 이벤트 중복 검사를 수행해야 한다.
- 체크아웃 생성 시 내부 사용자 ID, 구매 SKU, 예상 적립량을 함께 기록하고, 웹훅 수신 시 이 매핑과 일치할 때만 적립한다.
- 결제 시작 요청 자체도 남용 방지를 위해 요청 제한 대상에 포함한다.

## 수용 기준
- 신규 사용자는 최초 1회만 100 크레딧을 받는다.
- 생성 성공 또는 수정 성공 시 각 요청마다 잔액이 10 감소한다.
- 생성 또는 수정 최종 실패 시 해당 요청의 환불 이벤트가 기록되어 잔액이 복구된다.
- 잔액이 10 미만이면 생성과 수정 요청이 시작되지 않는다.
- 같은 Polar 결제 웹훅이 여러 번 도착해도 크레딧 적립은 한 번만 발생한다.
- 결제 완료 후 적립된 크레딧은 잔액 화면과 원장 이력에서 모두 확인 가능해야 한다.
- 서명 검증에 실패한 웹훅은 적립 효과를 만들지 않는다.
- 기록된 사용자 또는 SKU와 일치하지 않는 주문은 적립 처리되지 않는다.

## 영향 받는 영역
- 크레딧 원장.
- 잔액 표시 UI.
- 생성과 수정 요청 사전 검증.
- Polar 체크아웃 시작 API.
- Polar 웹훅 처리.

## 로컬 설계 메모
- 잔액은 캐시가 아니라 원장 합산 결과를 정본으로 삼는다.
- 웹훅 반영은 결제 성공 리다이렉트보다 신뢰도가 높은 비동기 경로를 기준으로 한다.
- SKU와 크레딧 수량은 운영 설정으로 관리하되, 적립 규칙은 항상 원장 이벤트로 남긴다.

## 실행 메타데이터
- 실행 단위: 문서, 코드, 결제 설정, 데이터베이스.
- 전제 조건: 인증된 사용자, Polar 상품 구성, SKU-크레딧 매핑, 원장 구조.
- 실행 후 확인 항목: 체크아웃 생성, 웹훅 반영, 잔액 갱신, 원장 이벤트 생성.
- 남겨야 할 증거: 주문 ID, 웹훅 이벤트 ID, 적립 이벤트 ID, 주문별 중복 방지 키.

## 위험과 열린 질문
- 출시 SKU 수와 가격은 별도 사업 결정이 필요하다.
- 결제 성공 리다이렉트가 먼저 도착하고 웹훅이 늦게 도착할 수 있으므로 UI 상태 설계가 중요하다.
- 악의적 결제 시작 반복 호출은 비용보다는 운영 소음을 유발하므로 최소한의 제한 정책이 필요하다.

## 검증 계획
- 무료 100 크레딧이 한 번만 지급되는지 확인한다.
- 생성과 수정 요청당 10 크레딧이 정확히 차감되는지 확인한다.
- 실패 시 환불 이벤트가 잔액을 복구하는지 확인한다.
- 같은 결제 웹훅 중복 수신 시 한 번만 적립되는지 확인한다.
- 서명 검증 실패 웹훅이 거절되는지 확인한다.
- 사용자 또는 SKU 바인딩이 어긋난 웹훅이 적립되지 않는지 확인한다.
