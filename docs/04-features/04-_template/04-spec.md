# 기능 스펙 템플릿

## 목적
- 하나의 기능에 대한 동작, 범위, 수용 기준, 로컬 설계 메모를 정의한다.

## 사용 순서
- 4단계: 기능 구현을 시작하기 전에 작성한다.
- 새로운 기능마다 이 템플릿을 복사해서 `docs/04-features/<feature>/04-spec.md`로 사용한다.

## 대상 독자
- 단일 기능을 구현, 검토, 테스트, 승인하는 사람.

## 범위 밖
- 전역 제품 전략.
- 전체 시스템 아키텍처.
- 저장소 전반의 규칙.
- 지속되는 아키텍처 이력.

## 정본으로 다루는 범위
- 기능 문제 정의.
- 기능 범위와 비목표.
- 사용자 또는 API 동작.
- 수용 기준.
- 영향 범위와 로컬 위험.

## 마지막 업데이트 시점
- 기능 범위, 동작, 수용 기준, 구현 접근이 실질적으로 바뀔 때.

## 정본 링크
- 제품 의도는 `docs/01-product.md`를 참고한다.
- 아키텍처 맥락은 `docs/02-system-architecture.md`를 참고한다.
- 저장소 전반의 규칙은 `docs/03-engineering-rules.md`를 참고한다.
- 이 기능이 오래 유지되는 결정에 의존한다면 관련 ADR을 참고한다.

## 문제
- 이 기능은 어떤 문제를 해결하는가?
- 왜 지금 이 기능이 필요한가?

## 고객 서사
- 이 기능이 들어가는 실제 사용자 흐름은 무엇인가?
- 사용자는 어떤 순간에 이 기능을 필요로 하는가?
- 기능이 없을 때 사용자가 겪는 대표적인 불편 또는 실패 경험은 무엇인가?
- 기능이 잘 동작했을 때 사용자가 체감해야 하는 변화는 무엇인가?

## 범위
- 이 기능에 포함되는 내용.
- 인증, 인가, 소유권 검증이 필요한 경계가 있다면 함께 적는다.
- 요청 제한, abuse 방지, 비용 유발 경로 보호가 필요한 경우 그 범위를 함께 적는다.

## 비목표
- 이 기능에 명시적으로 포함하지 않는 내용.

## 사용자 또는 호출자
- 누가 이 기능을 사용하는가?
- 주요 흐름은 무엇인가?
- 로그인 사용자, 비로그인 사용자, 운영자, 웹훅 처리자처럼 권한이 다른 호출자가 있다면 구분한다.

## 동작
- 기대되는 사용자 가시 동작 또는 API 가시 동작을 설명한다.
- 정확성에 실질적인 영향을 주는 엣지 케이스를 포함한다.
- 인증, 인가, 리소스 소유권, 비정상 접근 차단 규칙을 포함한다.
- 결제나 웹훅이 포함되면 서명 검증, replay 방지, 바인딩 검증 요구를 포함한다.
- 공개 엔드포인트이거나 비용 유발 기능이면 rate limit, 동시성 제한, abuse 방지 동작을 포함한다.
- Storage 또는 다운로드가 포함되면 공개 여부, 접근 경로, 만료 정책을 포함한다.

## 수용 기준
- 각 기준을 통과/실패로 판정 가능한 문장으로 작성한다.
- 구현 세부보다 관찰 가능한 결과를 우선한다.
- 권한이 없는 사용자, 잘못된 서명, 재사용 토큰, 과도한 요청 같은 부정 시나리오도 포함한다.

## 영향 받는 영역
- 영향을 받을 가능성이 높은 코드 경로, 시스템, 데이터, 운영 영역을 적는다.
- 인증 세션, 권한 정책, Storage 접근, 공개 엔드포인트, 외부 웹훅 같은 보안 경계도 적는다.

## 로컬 설계 메모
- 이 기능에만 특화된 설계 세부만 포함한다.
- 전역 아키텍처를 다시 설명하지 말고 링크로 연결한다.
- 이 기능이 추가로 요구하는 보안 경계, 시크릿, 권한 모델, 로그 신호가 있다면 여기 적는다.

## 실행 메타데이터
- 이 기능 변경의 실행 단위는 무엇인가? 예: 문서, 코드, 데이터, 운영 설정.
- 적용 전에 반드시 확인해야 할 전제 조건은 무엇인가?
- 실행 후 바로 확인해야 할 로그, 지표, 화면, 응답은 무엇인가?
- 실행 흔적을 남겨야 한다면 어떤 증거를 남겨야 하는가?
- 필요한 시크릿, 권한, 서명 검증 정보, 요청 제한 설정이 있다면 적는다.

## 위험과 열린 질문
- 아직 정리되지 않은 위험, 미지수, 미결정 사항.
- abuse, DDoS, 권한 우회, 데이터 노출, 비용 증폭 위험이 있다면 반드시 적는다.

## 검증 계획
- 이 기능을 어떻게 검증할지 정의한다: 테스트, 수동 QA, 지표, 롤아웃 점검 등.
- 정상 흐름뿐 아니라 권한 거절, 서명 검증 실패, 재사용 토큰, rate limit 발동, ownership 위반 시나리오를 포함한다.
