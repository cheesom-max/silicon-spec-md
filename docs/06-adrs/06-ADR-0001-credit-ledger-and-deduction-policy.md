# ADR-0001: 크레딧 원장과 차감 정책

## 목적
- 크레딧 적립, 차감, 환불을 어떤 데이터 모델과 정책으로 유지할지 기록한다.

## 사용 순서
- 6단계: 오래 유지될 크레딧 정책을 정할 때 읽고 갱신한다.
- 결제, 차감, 환불 구조가 바뀌면 함께 수정한다.

## 대상 독자
- 결제, 원장, 잔액 계산 로직을 설계하고 검토하는 엔지니어와 AI 에이전트.

## 범위 밖
- 전체 시스템 구조 설명.
- 기능별 UI 동작.
- 운영 절차의 상세 단계.

## 정본으로 다루는 범위
- 크레딧 데이터 모델.
- 차감 시점.
- 환불 시점.
- 적립과 중복 방지 원칙.

## 마지막 업데이트 시점
- 잔액 계산 방식, 차감 타이밍, 환불 규칙이 바뀔 때.

## 정본 링크
- 현재 시스템 구조는 `docs/02-system-architecture.md`를 참고한다.
- 관련 기능 스펙은 `docs/04-features/credits-and-billing/04-spec.md`를 참고한다.

## 상태
- 수락됨

## 맥락
- 이 제품은 무료 지급, 생성 차감, 수정 차감, 실패 환불, 결제 적립이 서로 다른 이벤트로 발생한다.
- 단순 잔액 필드만 두면 중복 적립, 누락 환불, 잘못된 동시성 업데이트를 추적하기 어렵다.

## 결정
- 잔액의 정본은 변경 가능한 단일 숫자 필드가 아니라 불변 원장 이벤트의 합산 결과로 둔다.
- 원장 이벤트 유형은 최소한 `grant_free`, `purchase_credit`, `spend_generate`, `spend_revision`, `refund_generate`, `refund_revision`을 포함한다.
- 생성과 수정 요청은 서버가 작업을 수락하는 시점에 각각 10 크레딧을 차감한다.
- 작업이 최종 실패하면 대응되는 환불 이벤트를 기록해 원장을 상쇄한다.
- Polar 주문 적립은 외부 주문 ID 또는 웹훅 이벤트 ID를 기준으로 idempotent 하게 처리한다.

## 실행 메타데이터
- 직접 영향 범위: 잔액 계산, 생성/수정 요청 사전 검증, 결제 웹훅 처리, 운영 정합성 점검.
- 적용 전 전제 조건: 사용자 식별, 주문 식별, 작업 식별이 안정적으로 존재해야 한다.
- 적용 이후 핵심 신호: 잔액 불일치 건수, 중복 적립 건수, 환불 누락 건수.

## 결과
- 긍정적 결과
  - 적립, 차감, 환불 이력이 모두 남아 감사를 쉽게 한다.
  - 결제 웹훅 중복 수신이나 생성 실패 후 복구를 안정적으로 처리할 수 있다.
  - 운영자가 특정 사용자 잔액을 재계산할 수 있다.
- 부정적 결과
  - 단순 잔액 필드보다 조회와 모델 설계가 복잡해진다.
  - 원장 이벤트와 작업 이벤트의 관계를 정확히 관리해야 한다.
- 후속 작업
  - 원장 정합성 점검 런북을 유지한다.
  - 주문 중복 방지 키 정책을 구현에 반영한다.

## 고려한 대안
- 사용자별 가변 잔액 필드만 유지하는 방안
  - 단순하지만 감사와 복구가 어렵고 외부 이벤트 중복 처리에 취약해 채택하지 않았다.
- 결제 적립만 원장으로 두고 차감은 잔액 필드로 처리하는 혼합 방안
  - 규칙 일관성이 깨지고 디버깅 비용이 높아 채택하지 않았다.
