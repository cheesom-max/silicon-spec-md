# ADR-0002: 무문자 출력 강제 정책

## 목적
- 생성 결과 이미지에 텍스트가 포함되지 않도록 어떤 정책으로 강제할지 기록한다.

## 사용 순서
- 6단계: 생성 품질의 핵심 정책을 정할 때 읽고 갱신한다.
- 무문자 보장 방식이나 실패 처리 정책이 바뀔 때 수정한다.

## 대상 독자
- 생성 품질, 검출 가드, 실패 처리 정책을 설계하고 검토하는 엔지니어와 AI 에이전트.

## 범위 밖
- 전체 생성 모델 선택 배경.
- 텍스트 오버레이 UI의 세부 동작.
- 운영 절차 상세.

## 정본으로 다루는 범위
- 무문자 정책의 강제 방식.
- 검출 실패 처리.
- 사용자 노출 조건.

## 마지막 업데이트 시점
- 검출 방식, 재시도 정책, 실패 기준이 바뀔 때.

## 정본 링크
- 현재 시스템 구조는 `docs/02-system-architecture.md`를 참고한다.
- 관련 기능 스펙은 `docs/04-features/thumbnail-generation/04-spec.md`를 참고한다.

## 상태
- 수락됨

## 맥락
- 이 제품의 핵심 약속은 자동 생성 이미지 안에 텍스트가 없다는 점이다.
- 생성 프롬프트만으로는 텍스트 유입을 완전히 막기 어렵고, 특히 과장된 썸네일 스타일에서는 모델이 임의의 문자를 넣을 수 있다.

## 결정
- 무문자 정책은 두 단계로 강제한다.
  - 첫째, 생성 프롬프트와 시스템 지시에서 텍스트, 로고형 문자, 임의 문자를 강하게 금지한다.
  - 둘째, 생성 결과를 텍스트 검출 가드에 통과시킨 뒤에만 완료 상태로 사용자에게 노출한다.
- 첫 결과가 검출에 실패하면 내부 재생성을 한 번만 수행한다.
- 재생성 후에도 실패하면 작업을 실패 처리하고 사용자가 지불한 10 크레딧을 환불한다.
- 사용자가 직접 추가하는 텍스트 오버레이는 이 정책의 예외가 아니라 별도 후처리 기능으로 본다. 즉, 원본 생성 자산은 항상 무문자여야 한다.

## 실행 메타데이터
- 직접 영향 범위: 생성 완료 판정, 실패 환불, 품질 지표, 운영 모니터링.
- 적용 전 전제 조건: 텍스트 검출 가드가 존재해야 하고, 실패 시 환불 경로가 구현되어 있어야 한다.
- 적용 이후 핵심 신호: 검출 실패율, 재생성 성공률, 사용자 불만 건수.

## 결과
- 긍정적 결과
  - 제품 핵심 약속을 기술적으로 강제할 수 있다.
  - 텍스트가 포함된 결과를 완료 상태로 노출할 가능성을 낮춘다.
  - 실패 시 환불 정책이 명확해져 사용자 신뢰가 높아진다.
- 부정적 결과
  - 생성 지연이 늘어날 수 있다.
  - 검출 민감도가 지나치면 정상 이미지를 차단할 수 있다.
- 후속 작업
  - 검출 실패 샘플을 주기적으로 검토한다.
  - 재생성 1회 정책이 충분한지 운영 데이터를 본 뒤 재평가한다.

## 고려한 대안
- 프롬프트 지시만으로 무문자 정책을 처리하는 방안
  - 구현은 단순하지만 완성도와 신뢰도가 낮아 채택하지 않았다.
- 사람이 수동 검토한 뒤 완료 처리하는 방안
  - 품질은 높일 수 있으나 MVP의 속도와 비용 구조에 맞지 않아 채택하지 않았다.
