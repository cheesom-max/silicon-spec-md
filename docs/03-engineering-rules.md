# Engineering Rules Template

## Purpose
- Define repository-wide engineering invariants and conventions.

## Usage Order
- Step 3: Write this after product and architecture are organized.
- Use it when the team is locking in shared rules before implementation starts.

## Intended Audience
- Engineers and AI agents who write, review, and maintain work in this repository.

## Out of Scope
- Product goals.
- Architecture explanations.
- Feature-level behavior.
- Operational incident response procedures.

## Canonical Scope
- Required patterns.
- Forbidden patterns.
- Naming and structure rules.
- Review and verification rules.

## Last Updated When
- When repository-wide rules change or repeated review comments should be promoted into standing rules.

## Canonical Links
- For architectural context, see `docs/02-system-architecture.md`.
- For project-level quality strategy, see `docs/05-test-strategy.md`.

## Design Principles
- A small set of core principles that guide implementation choices.

## Required Patterns
- Patterns that must be followed unless an ADR explicitly allows an exception.

## Forbidden Patterns
- Patterns that contributors must avoid.

## Naming and Structure Rules
- Rules for naming files, modules, directories, and symbols.

## Error Handling Rules
- Define what good error handling looks like in this repository.

## Test Rules
- Define the minimum contributor obligations for tests, evidence, and verification.
- Project-level risk strategy and exit criteria belong in `docs/05-test-strategy.md`.

## Review Rules
- Conditions that must be satisfied before work is considered ready for review.

## Decision Heuristics
- Short decision rules for common trade-offs.

## Exceptions
- Define how justified exceptions should be documented and approved.
