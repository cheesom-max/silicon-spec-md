# 엔지니어링 규칙

## 목적
- 구현이 시작될 때 따라야 할 저장소 전반의 불변 규칙과 관례를 정의한다.

## 사용 순서
- 3단계: 제품과 아키텍처를 정리한 뒤 읽고 수정한다.
- 구현을 시작하기 전에 팀 공통 규칙을 확정할 때 사용한다.

## 대상 독자
- 이 저장소에서 코드를 작성, 검토, 유지보수하는 엔지니어와 AI 에이전트.

## 범위 밖
- 제품 목표의 우선순위 판단.
- 시스템 구조의 상세 설명.
- 기능별 사용자 동작.
- 운영 사고 대응 절차.

## 정본으로 다루는 범위
- 필수 패턴.
- 금지 패턴.
- 네이밍 및 구조 규칙.
- 에러 처리, 테스트, 리뷰 규칙.

## 마지막 업데이트 시점
- 저장소 전반의 규칙이 바뀌거나 반복 리뷰 코멘트를 상시 규칙으로 올려야 할 때.

## 정본 링크
- 아키텍처 맥락은 `docs/02-system-architecture.md`를 참고한다.
- 프로젝트 수준 품질 전략은 `docs/05-test-strategy.md`를 참고한다.

## 설계 원칙
- 서버 권위를 우선한다. 크레딧, 결제, 생성 완료 여부는 브라우저가 아니라 서버가 최종 판정한다.
- 단일 정본을 지킨다. 잔액, 작업 상태, 자산 메타데이터는 각각 하나의 정본 테이블과 문서에서만 소유한다.
- 실패는 명시적으로 드러낸다. 생성 실패, 웹훅 실패, 텍스트 검출 실패는 조용히 숨기지 않는다.
- MVP 범위를 지킨다. 필요한 동작만 만들고 범용 편집기나 협업 기능으로 확장하지 않는다.

## 필수 패턴
- 모든 크레딧 변화는 원장 이벤트로 기록한다.
- 결제 웹훅 처리는 idempotent 해야 한다.
- Supabase 데이터 접근은 RLS 정책을 기본으로 설계한다.
- 외부 서비스 시크릿은 서버 런타임에서만 접근한다.
- 이미지 생성 결과는 사용자에게 완료 상태로 노출하기 전에 무문자 검출 게이트를 통과해야 한다.
- 수동 문구 오버레이는 AI 생성과 분리된 흐름으로 유지한다.
- 매직 링크 로그인은 허용된 리다이렉트 경로만 사용하고, 만료 시간과 재사용 방지 조건을 명시적으로 적용한다.
- Polar 웹훅은 서명 검증, 이벤트 중복 방지, 주문-사용자-SKU 매핑 검증을 모두 통과해야 적립 처리할 수 있다.
- 서비스 롤 키 사용 경로는 최소화하고, 사용자 요청 경로에서는 공개 키 + RLS 또는 서버 소유권 검증을 기본값으로 사용한다.
- 가입, 생성, 수정, 결제 시작 같은 공개 또는 비용 유발 엔드포인트에는 최소한의 rate limit와 abuse 방지 규칙을 둔다.
- Storage 객체 경로는 소유자 식별자와 연결되고, 다른 사용자의 객체를 직접 추측해 접근할 수 없어야 한다.

## 금지 패턴
- 클라이언트에서 잔액을 직접 증가 또는 감소시키는 패턴.
- 결제 성공을 브라우저 리다이렉트만으로 확정하는 패턴.
- 웹훅 중복 수신을 고려하지 않는 적립 로직.
- 무문자 요구사항을 프롬프트 문구만으로 해결했다고 가정하는 패턴.
- 텍스트 오버레이 기능을 범용 캔버스 편집기로 확장하는 패턴.
- 서명 검증 없이 웹훅 본문만 보고 적립 처리하는 패턴.
- 공개 Storage URL을 장기 정본 접근 경로로 사용하는 패턴.
- 로그인 여부만 확인하고 자산 소유권을 별도로 검증하지 않는 패턴.
- 무료 크레딧 지급, 가입, 생성 요청에 요청 제한을 두지 않는 패턴.

## 네이밍 및 구조 규칙
- 기능 스펙은 `docs/04-features/<feature>/04-spec.md` 형식을 유지한다.
- 기능 디렉토리는 케밥 케이스를 사용한다.
- 미래 코드 저장소가 생기면 서버 측 결제, 크레딧, 생성, 편집 모듈을 분리해 경계를 유지한다.
- 데이터베이스 테이블 이름은 역할이 분명한 복수형 또는 도메인 명사형을 사용한다.

## 에러 처리 규칙
- 사용자가 조치할 수 있는 오류는 원인과 다음 행동을 함께 보여준다.
- 크레딧 부족은 요청 거절과 함께 현재 부족 상태를 명확히 알려준다.
- 생성 실패 후 환불이 발생하면 실패 원인과 환불 상태를 함께 남긴다.
- 재시도 가능한 외부 실패와 즉시 중단해야 하는 검증 실패를 구분한다.
- 인증 또는 권한 오류는 존재 여부를 과도하게 노출하지 않는 범위에서 일관된 메시지로 응답한다.
- rate limit 또는 abuse 차단은 거절 사유를 남기되, 내부 탐지 기준의 상세 수치는 외부에 노출하지 않는다.

## 테스트 규칙
- 크레딧 차감, 환불, 적립은 단위 테스트와 통합 테스트 모두에서 검증한다.
- 사용자별 데이터 격리는 RLS 또는 권한 경로에서 검증한다.
- 무문자 정책은 생성 성공 테스트와 실패 테스트를 모두 가져야 한다.
- 프로젝트 수준 위험 전략과 종료 기준은 `docs/05-test-strategy.md`를 따른다.
- 웹훅 서명 검증 실패, 재생 공격, 잘못된 주문-사용자 바인딩은 부정 테스트로 검증한다.
- signup, generation, checkout 시작 경로의 rate limit과 동시성 제한은 통합 테스트 또는 운영 검증 시나리오로 확인한다.

## 리뷰 규칙
- 기능 변경은 관련 기능 스펙과 일치해야 한다.
- 결제 또는 크레딧 로직 변경은 관련 ADR과 런북 영향까지 함께 검토한다.
- 새 외부 의존성을 추가하면 시스템 아키텍처 문서 반영 여부를 함께 확인한다.
- 사용자에게 노출되는 상태 이름과 실패 메시지는 제품 의도와 맞아야 한다.
- 인증, 결제, 스토리지, 다운로드 경로 변경은 보안 경계와 운영 런북 영향까지 함께 검토한다.

## 결정 휴리스틱
- 브라우저 편의성과 원장 정합성이 충돌하면 원장 정합성을 우선한다.
- 기능 확장성과 MVP 속도가 충돌하면 현재 제품 목표를 만족하는 가장 단순한 구조를 택한다.
- 자동화와 수동 검토가 충돌하면 결제와 크레딧 영역은 자동화 후 수동 점검 경로까지 남긴다.

## 예외
- 전역 규칙의 예외는 관련 ADR이나 기능 스펙에 이유와 범위를 함께 남긴다.
- 일회성 운영 우회는 런북에 기록하고 장기 정책으로 승격할지 검토한다.
