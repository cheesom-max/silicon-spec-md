# 테스트 전략

## 목적
- 이 프로젝트의 품질 위험과 검증 접근 방식을 정의한다.

## 사용 순서
- 5단계: 기능 스펙이 작성된 뒤 프로젝트 공통 검증 원칙을 정리할 때 사용한다.
- 저장소 전체에 공통으로 적용할 품질 기준이 필요할 때 읽고 수정한다.

## 대상 독자
- 변경 사항을 어떻게 검증해야 하는지 결정해야 하는 엔지니어와 AI 에이전트.

## 범위 밖
- 기능별 상세 테스트 케이스의 전체 목록.
- 사고 대응 절차.
- 제품 우선순위 결정.
- 상세 아키텍처 설명.

## 정본으로 다루는 범위
- 테스트 수준.
- 품질 위험.
- 요구되는 검증 증거.
- 변경 사항의 종료 기준.

## 마지막 업데이트 시점
- 검증 모델, 테스트 도구, 위험 프로파일, 릴리스 게이트가 바뀔 때.

## 정본 링크
- 저장소 전반의 규칙은 `docs/03-engineering-rules.md`를 참고한다.
- 기능별 검증 계획은 각 기능 스펙을 참고한다.

## 범위
- 인증, 크레딧 원장, 썸네일 생성, 수정, 텍스트 오버레이, 결제 웹훅, 사용자 자산 접근 제어를 다룬다.

## 위험 영역
- 무료 100 크레딧이 중복 지급되는 실패.
- 생성 또는 수정 실패 후 환불이 누락되는 실패.
- Polar 웹훅 중복 수신으로 중복 적립이 발생하는 실패.
- 사용자가 다른 사용자의 생성 결과나 결제 이력을 접근하는 실패.
- 완료 상태 결과물에 텍스트가 포함된 채 노출되는 실패.
- 서명 검증이 되지 않은 웹훅이 적립 효과를 만드는 실패.
- signup farming, generation abuse, checkout spam이 비용 또는 가용성 문제로 이어지는 실패.
- 트래픽 급증 시 생성 경로가 무제한으로 외부 모델 비용을 증폭시키는 실패.

## 테스트 수준
- 단위 테스트
  - 크레딧 차감, 적립, 환불 계산 규칙.
  - 생성 상태 전이 규칙.
  - 텍스트 검출 결과에 따른 완료 또는 실패 판정.
- 통합 테스트
  - 인증 후 프로필 생성과 초기 크레딧 지급.
  - 생성 요청과 원장 차감, 실패 시 환불.
  - Polar 웹훅과 idempotent 적립 처리.
  - Storage 접근 권한과 자산 소유권 검사.
  - Polar 웹훅 서명 검증과 사용자-SKU 바인딩 검사.
  - signup, generation, checkout 시작 경로의 요청 제한과 동시성 제한.
- 종단간 테스트
  - 신규 가입 후 첫 썸네일 생성 성공.
  - 크레딧 부족 상태에서 생성 차단.
  - 기존 생성본 수정 후 새 버전 저장.
  - 결제 후 크레딧 증가 반영.
  - 문구 오버레이 저장 및 다운로드.
- 수동 QA
  - 생성 흐름에서 진행 상태와 오류 메시지의 이해 가능성.
  - 무문자 결과 이미지의 실제 확인.
  - 다운로드한 이미지의 화면 품질과 크기 확인.

## 핵심 시나리오
- 첫 가입 사용자는 정확히 한 번만 100 크레딧을 받는다.
- 30자 미만 설명은 생성 요청 전에 차단된다.
- 생성 성공 시 10 크레딧이 차감되고 결과가 저장된다.
- 생성 실패 시 사용자 잔액은 차감 전과 동일하게 복구된다.
- 수정 성공 시 기존 자산은 유지되고 새 버전이 추가된다.
- 수동 문구 오버레이는 크레딧 차감 없이 저장 또는 다운로드된다.
- 결제 완료 웹훅은 한 번만 적립 효과를 만든다.
- 잘못된 서명이나 잘못된 주문 바인딩을 가진 웹훅은 적립되지 않는다.
- 짧은 시간 안의 반복 가입 또는 생성 요청은 제한된다.

## 환경과 픽스처
- 테스트용 사용자 계정과 초기 잔액 픽스처가 필요하다.
- 텍스트가 없는 이미지와 텍스트가 포함된 이미지 두 종류의 검출 픽스처가 필요하다.
- 결제 웹훅 재전송과 중복 수신을 재현할 수 있는 테스트 이벤트가 필요하다.
- 생성 실패와 타임아웃을 강제로 만들 수 있는 외부 서비스 모의 환경이 필요하다.

## 검증 증거 기대치
- 기능별 상태 전이와 원장 이벤트가 로그 또는 데이터 조회로 확인되어야 한다.
- 잔액 화면과 원장 합산 결과가 일치해야 한다.
- 결제 적립은 웹훅 이벤트 ID와 내부 적립 이벤트가 연결되어야 한다.
- 무문자 정책은 검출 통과 또는 차단 이력이 남아야 한다.
- 요청 제한 또는 abuse 차단은 로그, 메트릭, 거절 응답으로 확인 가능해야 한다.
- 인증 및 세션 관련 보안 이벤트는 운영자가 추적 가능한 수준으로 남아야 한다.

## 종료 기준
- 관련 기능 스펙의 수용 기준이 모두 충족된다.
- 크레딧 적립, 차감, 환불 경로에 대한 자동 검증이 존재한다.
- 사용자 데이터 격리와 자산 접근 제어가 검증된다.
- 결제 또는 생성 실패 시 운영자가 복구 가능한 증거가 남는다.

## 운영 신호와 롤백 체크
- 고위험 변경에서는 생성 실패율, 환불 누락 건수, 중복 적립 건수, 텍스트 검출 차단 비율을 확인한다.
- 이상 징후는 잔액 불일치, 결제 후 미반영, 다른 사용자 자산 노출, 무문자 정책 위반으로 판단한다.
- 결제 적립이나 생성 차감이 잘못 반영되면 배포 진행을 멈추고 런북에 따라 복구 또는 롤백 절차로 전환한다.
- signup 실패율 급증, rate limit 발동 급증, 생성 요청 폭증, 서명 검증 실패 급증은 abuse 또는 트래픽 공격 신호로 본다.

## 공백과 예외
- 실제 구현 저장소가 아직 없으므로 구체적인 테스트 명령은 이후 코드 저장소에서 정의한다.
- 정당한 품질 예외가 필요하면 관련 기능 스펙과 런북에 기간, 영향 범위, 보완 수단을 함께 남긴다.
