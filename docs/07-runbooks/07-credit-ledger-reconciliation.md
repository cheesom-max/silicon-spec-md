# 크레딧 원장 정합성 점검 런북

## 목적
- 사용자 표시 잔액과 원장 이벤트 합산 결과가 일치하지 않을 때 점검하고 복구하는 절차를 제공한다.

## 사용 순서
- 7단계: 잔액 불일치, 환불 누락, 중복 차감 의심이 있을 때 사용한다.
- 원장 구조나 보정 정책이 바뀌면 갱신한다.

## 대상 독자
- 잔액 이상 징후를 분석하고 복구해야 하는 운영자와 엔지니어.

## 범위 밖
- 제품 우선순위 결정.
- 상세 아키텍처 설명.
- 결제 공급자 자체 장애 분석.

## 정본으로 다루는 범위
- 트리거 조건.
- 사전 조건.
- 실행 절차.
- 검증, 복구, 에스컬레이션.

## 마지막 업데이트 시점
- 원장 구조, 보정 정책, 운영 점검 절차가 바뀔 때.

## 정본 링크
- 관련 기능 스펙은 `docs/04-features/credits-and-billing/04-spec.md`를 참고한다.
- 관련 ADR은 `docs/06-adrs/06-ADR-0001-credit-ledger-and-deduction-policy.md`를 참고한다.

## 트리거
- 사용자 화면 잔액과 내부 계산 결과가 다르게 보이는 경우.
- 생성 실패 후 환불되지 않았다는 신고가 들어온 경우.
- 중복 차감 또는 중복 적립이 의심되는 경우.

## 사전 조건
- 사용자 ID와 의심 구간의 시간 범위를 확보해야 한다.
- 원장 이벤트, 생성 작업, 결제 주문을 조회할 수 있어야 한다.

## 필요한 접근 권한과 도구
- 데이터베이스 조회 권한.
- 애플리케이션 로그 조회 권한.
- 필요 시 수동 보정 이벤트 작성 권한.

## 실행 메타데이터
- 적용 범위: 특정 사용자 또는 특정 기간의 원장 이력.
- 실행 전 기록할 기준 상태: 사용자 현재 표시 잔액, 원장 이벤트 수, 관련 작업 수, 관련 주문 수.
- 실행 중 확인할 신호: 작업별 차감-환불 짝, 주문별 적립 중복 여부, 시각 표시 잔액 계산 로직.

## 절차
1. 사용자 ID와 문제가 발생한 시간 범위를 기준으로 원장 이벤트를 조회한다.
2. 이벤트를 유형별로 분류해 무료 지급, 결제 적립, 생성 차감, 수정 차감, 환불 이벤트를 구분한다.
3. 생성 및 수정 작업 이력과 대조해 차감 이벤트마다 대응되는 작업이 있는지 확인한다.
4. 실패 작업에 대응되는 환불 이벤트가 누락되었는지 확인한다.
5. Polar 주문 이력과 대조해 적립 이벤트가 누락되었거나 중복되었는지 확인한다.
6. 불일치 원인을 특정한 뒤, 필요한 경우 단일 보정 이벤트를 추가해 원장을 정정한다.
7. 표시 잔액 계산 경로가 원장 합산 결과와 동일한지 확인한다.

## 검증
- 보정 후 사용자 표시 잔액과 원장 합산 결과가 일치해야 한다.
- 같은 작업 또는 주문에 대해 중복 보정이 남지 않아야 한다.
- 복구 내역과 원인이 운영 기록으로 남아 있어야 한다.

## 롤백 또는 복구
- 잘못된 보정 이벤트를 넣었다면 상쇄 이벤트를 추가해 원장을 원상 복구한다.
- 시스템적 계산 오류가 확인되면 추가 보정을 멈추고 계산 로직 수정 작업으로 전환한다.

## 에스컬레이션
- 하루 안에 동일 유형의 정합성 이슈가 3건 이상 발생하면 결제/원장 담당 엔지니어에게 에스컬레이션한다.
- 사용자 금전 손실 가능성이 있으면 즉시 운영 책임자에게 사용자 ID, 영향 범위, 임시 조치 내용을 공유한다.

## 운영 피드백 루프
- 모든 보정 작업 뒤에 원인 분류를 남긴다.
- 실패 패턴이 반복되면 기능 스펙, ADR, 테스트 전략까지 함께 수정한다.
- 필요하면 `docs/05-test-strategy.md`의 위험 영역과 검증 증거 기대치를 강화한다.
