# Polar 웹훅 실패 복구 런북

## 목적
- Polar 결제 완료 후 크레딧이 반영되지 않거나 중복 반영된 상황에 대응하는 반복 가능한 절차를 제공한다.

## 사용 순서
- 7단계: 운영 중 결제 반영 장애가 발생했을 때 사용한다.
- 웹훅 경로, 중복 방지 정책, 결제 반영 구조가 바뀌면 갱신한다.

## 대상 독자
- 결제 반영 장애를 실제로 확인하고 복구해야 하는 운영자와 엔지니어.

## 범위 밖
- 제품 기획.
- 세금 처리 이론.
- 전체 아키텍처 판단 근거.

## 정본으로 다루는 범위
- 트리거 조건.
- 사전 조건.
- 실행 절차.
- 검증, 복구, 에스컬레이션.

## 마지막 업데이트 시점
- 결제 공급자, 웹훅 이벤트, 복구 절차가 바뀔 때.

## 정본 링크
- 관련 기능 스펙은 `docs/04-features/credits-and-billing/04-spec.md`를 참고한다.
- 관련 ADR은 `docs/06-adrs/06-ADR-0001-credit-ledger-and-deduction-policy.md`를 참고한다.

## 트리거
- 사용자가 결제를 완료했는데 크레딧이 반영되지 않았다고 신고한 경우.
- 같은 주문이 두 번 이상 적립된 흔적이 보이는 경우.
- Polar 웹훅 전달 실패 또는 지연 알림을 받은 경우.

## 사전 조건
- Polar 대시보드 접근 권한.
- 애플리케이션 로그와 데이터베이스 조회 권한.
- 사용자 ID 또는 주문 ID를 확보해야 한다.

## 필요한 접근 권한과 도구
- Polar 관리자 대시보드.
- 배포 환경 로그 조회 도구.
- 데이터베이스 조회 및 수동 원장 조정 권한.

## 실행 메타데이터
- 적용 범위: 프로덕션 결제 반영 경로.
- 실행 전 기록할 기준 상태: 사용자 현재 잔액, 관련 주문 ID, 관련 웹훅 이벤트 ID, 기존 원장 이벤트 수.
- 실행 중 확인할 신호: 웹훅 수신 여부, 중복 방지 키 존재 여부, 적립 이벤트 생성 여부.

## 절차
1. 사용자 신고 또는 모니터링 알림에서 주문 ID와 사용자 ID를 확인한다.
2. Polar 대시보드에서 해당 주문이 실제로 `paid` 상태인지 확인한다.
3. 애플리케이션 로그에서 해당 주문 ID 또는 웹훅 이벤트 ID가 수신되었는지 확인한다.
4. 데이터베이스에서 같은 주문 ID로 기록된 적립 이벤트가 이미 존재하는지 조회한다.
5. 적립 이벤트가 없고 주문이 실제 `paid` 상태라면, 누락 원인을 확인한 뒤 단일 적립 이벤트를 수동 반영한다.
6. 적립 이벤트가 중복으로 존재하면 사용자 잔액 영향 범위를 계산하고 초과 적립분을 상쇄하는 보정 이벤트를 기록한다.
7. 원인을 기록하고, 재발 방지를 위해 웹훅 서명 검증, 재시도, 중복 방지 키 저장 상태를 점검한다.

## 검증
- 최종 잔액이 기대값과 일치해야 한다.
- 같은 주문 ID에 대해 적립 효과는 한 번만 남아 있어야 한다.
- 복구 후 사용자에게 확인 가능한 잔액 이력이 남아 있어야 한다.

## 롤백 또는 복구
- 수동 적립이 잘못 반영되면 동일 참조 ID를 가진 상쇄 보정 이벤트를 추가한다.
- 시스템적 장애가 지속되면 자동 적립을 일시 중단하고 수동 확인 모드로 전환한다.

## 에스컬레이션
- 동일 원인으로 하루 2건 이상 반복되면 결제 연동 담당 엔지니어에게 에스컬레이션한다.
- 사용자 금전 이슈가 포함되면 운영 책임자에게 주문 ID, 사용자 ID, 원장 보정 내역과 함께 보고한다.

## 운영 피드백 루프
- 복구 후 장애 원인과 보정 내역을 남긴다.
- 중복 적립 또는 미적립 패턴이 반복되면 ADR 또는 기능 스펙 업데이트를 검토한다.
- 필요하면 `docs/04-features/credits-and-billing/04-spec.md`와 관련 ADR을 함께 수정한다.
