# 런북 템플릿

## 목적
- 특정 작업이나 사고에 대해 반복 가능한 운영 절차를 제공한다.

## 사용 순서
- 7단계: 반복 수행해야 하는 운영 절차가 생겼을 때 작성한다.
- 배포, 장애 대응, 복구처럼 실제 실행 순서가 중요한 작업에 사용한다.

## 대상 독자
- 평시 또는 긴장 상황에서 실제로 절차를 수행하는 사람.

## 범위 밖
- 제품 기획.
- 아키텍처 판단 근거.
- 저장소 전반의 엔지니어링 정책.
- 장문의 트러블슈팅 이론.

## 정본으로 다루는 범위
- 트리거 조건.
- 사전 조건.
- 실행 절차.
- 검증, 복구, 에스컬레이션.

## 마지막 업데이트 시점
- 운영 절차가 바뀌거나, 도구가 바뀌거나, 실제 실행에서 빈틈이 드러날 때.

## 정본 링크
- 관련 ADR, 기능 스펙, 대시보드, 아키텍처 문서를 연결한다.

## 트리거
- 어떤 사건 또는 조건이 이 런북 사용을 요구하는가?
- 인증 이상 징후, 권한 우회, 웹훅 위조 의심, rate limit 급증, abuse 또는 DDoS 같은 보안 트리거가 있다면 명시한다.

## 사전 조건
- 실행 전에 반드시 충족되어야 하는 조건은 무엇인가?
- 필요한 계정 식별자, 주문 ID, 자산 ID, 로그 범위, 권한 경계를 함께 적는다.

## 필요한 접근 권한과 도구
- 필요한 계정, 권한, 대시보드, 명령어를 적는다.
- 세션 무효화, 요청 제한 조정, 서명 검증 상태 확인, Storage 접근 점검 권한이 필요하다면 적는다.

## 실행 메타데이터
- 이 절차가 적용되는 환경 또는 범위는 어디인가?
- 실행 전 기록해야 할 기준 상태는 무엇인가?
- 실행 중 반드시 확인해야 할 신호는 무엇인가?
- 보안 사고라면 인증 이벤트, 권한 거절 로그, 서명 검증 실패, rate limit 발동량 같은 신호를 포함한다.

## 절차
1. 첫 번째 단계.
2. 두 번째 단계.
3. 작업이 완료될 때까지 계속한다.

## 검증
- 절차가 성공했는지 어떻게 확인할 것인가?
- 보안 사고라면 재발 차단, 권한 정상화, 비용 안정화, 사용자 영향 범위 고정 여부를 포함한다.

## 롤백 또는 복구
- 절차가 실패하거나 되돌려야 할 때 무엇을 해야 하는가?
- 과도한 차단이나 잘못된 보정이 있었다면 단계적 완화 또는 상쇄 절차를 함께 적는다.

## 에스컬레이션
- 언제, 누구에게, 어떤 정보를 가지고 에스컬레이션할 것인가?
- 보안, 인증, 결제, 가용성 이슈별로 누가 오너인지와 어떤 식별자를 함께 넘길지 적는다.

## 운영 피드백 루프
- 이 런북을 실행한 뒤 반드시 남겨야 할 관찰 결과는 무엇인가?
- 어떤 반복 실패 패턴이 보이면 문서나 절차를 수정해야 하는가?
- 후속 수정이 필요할 때 어떤 문서까지 업데이트해야 하는가? 예: 기능 스펙, ADR, 테스트 전략.
- 반복되는 보안 트리거나 abuse 패턴이 보이면 관련 기능 스펙, ADR, 테스트 전략, 엔지니어링 규칙까지 함께 갱신한다.
