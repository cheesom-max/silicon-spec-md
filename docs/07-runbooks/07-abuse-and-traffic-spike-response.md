# abuse 또는 트래픽 급증 대응 런북

## 목적
- signup farming, generation abuse, checkout spam, layer-7 트래픽 급증이 발생했을 때 서비스 비용과 가용성 영향을 줄이는 대응 절차를 제공한다.

## 사용 순서
- 7단계: abuse 또는 트래픽 급증 정황이 발생했을 때 사용한다.
- 요청 제한 정책, 동시성 제한, 모니터링 지표가 바뀌면 갱신한다.

## 대상 독자
- rate limit, 생성 경로, 운영 모니터링을 실제로 조정해야 하는 운영자와 엔지니어.

## 범위 밖
- 장기적인 보안 설계 이론.
- 인증 침해 사고 대응.
- 결제 미반영 복구.

## 정본으로 다루는 범위
- 트리거 조건.
- 사전 조건.
- 실행 절차.
- 검증, 복구, 에스컬레이션.

## 마지막 업데이트 시점
- rate limit 정책, 트래픽 보호 계층, 생성 큐 정책이 바뀔 때.

## 정본 링크
- 관련 ADR은 `docs/06-adrs/06-ADR-0003-public-endpoint-security-baseline.md`를 참고한다.
- 관련 테스트 전략은 `docs/05-test-strategy.md`를 참고한다.

## 트리거
- signup 요청, 생성 요청, 결제 시작 요청이 평소보다 급격히 증가한 경우.
- 외부 모델 비용 또는 실패율이 비정상적으로 상승한 경우.
- rate limit 발동량이나 동시성 초과 건수가 급증한 경우.

## 사전 조건
- 요청량, 거절량, 생성 동시성, 외부 모델 오류율을 볼 수 있어야 한다.
- 운영자가 요청 제한 계층 또는 기능 플래그를 조정할 수 있어야 한다.

## 필요한 접근 권한과 도구
- 배포 환경 모니터링 도구.
- 애플리케이션 로그 조회 도구.
- 요청 제한 정책 조정 권한.
- 필요 시 생성 기능 임시 제한 권한.

## 실행 메타데이터
- 적용 범위: 가입, 로그인 요청, 생성, 수정, 결제 시작, 웹훅 수신 경로.
- 실행 전 기록할 기준 상태: 현재 요청량, 거절률, 생성 동시성, 외부 모델 실패율, 최근 비용 지표.
- 실행 중 확인할 신호: 특정 식별자 집중 요청, 글로벌 급증, 체크아웃 spam, queue 포화.

## 절차
1. 급증한 엔드포인트와 영향을 받는 사용자 흐름을 확인한다.
2. 특정 식별자 중심 abuse인지, 전체적인 트래픽 급증인지 구분한다.
3. 우선 해당 경로의 rate limit 또는 동시성 제한을 상향 또는 강화한다.
4. 생성 경로가 비용 증폭을 일으키면 사용자별 동시 작업 수를 더 낮추거나 일시적으로 새 요청을 지연 또는 차단한다.
5. signup farming이 의심되면 무료 크레딧 지급 패턴과 신규 계정 생성 패턴을 함께 조회한다.
6. checkout spam이 의심되면 결제 시작 경로의 요청 제한과 로그를 점검한다.
7. 상황이 안정되면 원인, 적용한 제한, 영향 범위를 기록한다.

## 검증
- 요청량 급증이 제어되고 생성 경로의 포화가 완화되어야 한다.
- 외부 모델 비용 증가 속도가 정상 범위로 돌아와야 한다.
- 합법 사용자 흐름이 완전히 차단되지 않았는지 확인해야 한다.

## 롤백 또는 복구
- 임시 제한이 과도했다면 단계적으로 완화한다.
- 문제 원인이 특정 식별자라면 전역 제한보다 국소 제한으로 복귀한다.

## 에스컬레이션
- 서비스 전반 가용성 저하가 발생하면 운영 책임자에게 즉시 에스컬레이션한다.
- 비용 손실이 빠르게 누적되면 제품 책임자와 운영 책임자에게 함께 공유한다.

## 운영 피드백 루프
- 급증 원인을 signup farming, generation abuse, checkout spam, 일반 트래픽 급증으로 분류해 기록한다.
- 반복 패턴이 보이면 ADR, 엔지니어링 규칙, 테스트 전략을 함께 수정한다.
